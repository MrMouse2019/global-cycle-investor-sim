# 结构化头脑风暴

## 1. 背景

源设计文档已经明确产品方向：轻策略、强教育、高真实度，核心是让新手理解“先看宏观 β、再选市场、最后挑板块个股”。本次头脑风暴用于把完整设想收敛为可落地 MVP，避免范围失控，并补齐长期周期、提前清仓、统计输出、合规与脱敏规则。

## 2. 核心问题与结论

### 2.1 MVP 是否保留完整长期周期？

结论：MVP 标准完整周期改为至少 20 年。

原因：

- 长期投资教育必须覆盖多轮牛熊、风格切换和复利路径。
- 结局体系需要足够长的行为数据，才能区分“长期能力”“短期运气”和“风险幸存”。
- 20 年可以更好体现盈利年份、亏损年份、最大回撤、年化收益和提前清仓风险。

执行方式：

- 每年仍保持一屏宏观报告、一屏配置、一屏结算复盘，避免 20 年流程过重。
- 年度报告文案保持短而明确。
- 允许玩家快速进入下一年。
- 如果触发清仓条件，可在 20 年前提前结束。

### 2.2 是否允许玩家提前结束游戏？

结论：允许，并作为核心风险教育机制。

提前结束不是随机死亡，而是由清晰规则触发：

- 总资产跌破清仓线。
- 最大回撤过深且风控评分极低。
- 连续多年重仓逆周期或单一赛道，并累计亏损严重。
- 长期高风险裸露，遇到风险事件时无分散或现金缓冲。

设计目的：

- 让玩家理解“活下来”是长期投资的前提。
- 避免玩家在前几年高风险暴赚后误以为策略正确。
- 把风控从评分项提升为生存条件。

### 2.3 首版是否做个股？

结论：MVP 不做真实个股，也不做可唯一映射真实公司的个股模式。

原因：

- 个股会显著增加内容、合规、脱敏和解释成本。
- 源设计的教育主线是宏观周期、市场差异和板块轮动，不依赖个股。
- 新手阶段先建立自上而下框架，比学习具体公司更重要。

执行方式：

- 首版只做市场 + 板块配置。
- 若需要增强代入感，可用“资产原型卡”，但名称必须脱敏且不含 ticker。

### 2.4 资金配置用自由输入还是预设模板？

结论：MVP 采用“预设模板 + 可调整权重”的折中方案。

原因：

- 完全自由输入容易增加新手负担。
- 只给模板会降低策略表达能力。
- 折中方式既能降低门槛，又能观察玩家是否重仓、分散或逆周期。

推荐模板：

- 稳健分散。
- 成长进攻。
- 防御避险。
- 周期资源。
- 现金观望。
- 自定义。

### 2.5 随机事件如何控制？

结论：事件是“现实不确定性教学”，不是胜负骰子。

规则：

- 每年 1-2 个事件。
- 事件影响范围明确：市场、板块、风险偏好或波动。
- 事件影响幅度受限。
- 事件不能让明显错误策略长期胜出。
- 结算页必须解释事件影响。
- 如果事件触发清仓，必须解释清仓来自既有高风险暴露与事件冲击叠加，而不是单纯“黑天鹅秒杀”。

### 2.6 教学反馈做到什么密度？

结论：每年输出 3 类反馈。

1. 你做对了什么。
2. 你踩了什么坑。
3. 下一次应该怎么判断。

避免：

- 大段金融教材。
- 只给收益率不解释。
- 用专业术语堆叠但不翻译。

### 2.7 难度曲线如何设计？

结论：20 年分四段。

- 年 1-3：强引导，明确提示适配市场和板块。
- 年 4-8：半引导，提示宏观环境但不直接给答案。
- 年 9-15：弱引导，要求玩家独立判断多轮周期切换。
- 年 16-20：长期复利和风控压力测试，强调守住收益、控制回撤和避免尾部风险。

### 2.8 是否需要教学模式和自由模式？

结论：MVP 只做教学模式，自由模式后续再加。

原因：

- 教学模式最能验证核心教育价值。
- 自由模式需要更多事件组合、随机种子和内容池，适合 MVP 后扩展。

### 2.9 是否本地存档？

结论：使用 localStorage。

原因：

- 前端-only 足以支撑单机体验。
- 避免后端和账号系统过早引入。
- 支持继续游戏和重开一局。

### 2.10 最终统计应输出什么？

结论：最终页必须输出盈利和亏损统计，而不仅是结局标签。

必须展示：

- 初始本金。
- 最终资产。
- 总盈利/亏损金额。
- 总收益率。
- 年化收益率。
- 已完成年份。
- 盈利年份数量。
- 亏损年份数量。
- 最好年份。
- 最差年份。
- 平均盈利年份收益率。
- 平均亏损年份收益率。
- 最大回撤。
- 是否提前清仓与原因。

## 3. 方案完善后的产品形态

MVP 是一个本地可运行网页游戏：

1. 玩家点击开始，获得 100 万模拟本金。
2. 系统进入第 1 年，展示宏观环境和教学提示。
3. 玩家选择配置模板或调整市场/板块权重。
4. 系统根据宏观周期、市场匹配、板块景气、仓位分散、年度事件计算结果。
5. 年终展示收益、盈利/亏损金额、资产、回撤、评分和复盘。
6. 年度结算后检查是否触发提前清仓。
7. 若未清仓，重复到至少第 20 年。
8. 若清仓，立即进入提前清仓结局。
9. 根据长期收益、亏损统计、认知、风控和习惯生成不少于 10 种最终结局之一。

## 4. 规则引擎原则

- 强逻辑、弱随机。
- 参数化、可测试。
- 收益和扣分必须可解释。
- 鼓励周期匹配和分散配置。
- 惩罚长期重仓梭哈、逆周期操作和高波动无保护。
- 清仓机制必须由资产损失、回撤、风控和错误习惯共同驱动。
- 最终评价必须同时看收益、亏损、回撤和完成年份。

## 5. UI 体验原则

- 每年最多 3 个主要界面：报告、配置、复盘。
- 配置页避免同时展示过多输入框。
- 所有术语都给一句白话解释。
- 关键结果用“为什么”解释，而不是只展示数字。
- 结局页强调成长标签、统计证据和方法论，而不是单纯炫耀收益。
- 提前清仓页要清楚说明清仓原因，避免玩家感觉是随机失败。

## 6. 内容与脱敏结论

- 首版用户可见内容不出现真实股票名、ticker、logo。
- 市场名和行业板块名可以保留，因为它们是宏观分类，不是单一证券推荐。
- 若出现股票风格案例，使用近似但非真实名称，例如英伟大、苹国、特思拉、台晶电。
- 所有案例应表达为“教学抽象原型”，不能表达为真实投资标的。
- 前端数据中不得保留真实原型映射表。

## 7. 主要风险与应对

| 风险 | 应对 |
| --- | --- |
| 20 年流程过长 | 保持每年 3 屏以内，并支持快速推进 |
| 清仓像随机惩罚 | 清仓必须绑定资产、回撤和错误行为，并输出原因 |
| 变成金融知识幻灯片 | 每年必须有配置决策和反馈 |
| 规则太复杂无法调参 | MVP 只保留宏观、市场、板块、仓位、事件、清仓六类因子 |
| 随机性误导学习 | 限制 luck 和事件幅度，结算解释来源 |
| 被误解为投资建议 | 不接真实行情，不显示真实证券，显示免责声明 |
| 与 fundpilot 混线 | 独立目录、独立技术栈、独立文档、独立命名 |

## 8. 最终收敛方案

推荐方案：

- 新建独立 Vite React TypeScript 项目。
- 前端-only。
- 至少 20 年教学模式。
- 支持提前清仓结束。
- 市场 + 板块配置，不做真实个股。
- 使用本地 JSON/TS 静态配置驱动内容。
- 使用纯 TypeScript 规则引擎计算结果、清仓和统计。
- 最终结局不少于 10 个。
- 最终页输出盈利和亏损统计。
- 使用 localStorage 存档。
- 文档先行，代码后置。
