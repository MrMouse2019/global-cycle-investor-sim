# 开发计划

## 1. 原则

- 先文档对齐，再写代码。
- 先规则引擎，再 UI 打磨。
- 先本地可运行，再考虑部署。
- 先市场/板块级模拟，再考虑个股扩展。
- 不复用 `../fundpilot/` 的业务模型、数据源或真实投资辅助逻辑。
- 标准完整周期至少 20 年，并允许风控失控时提前清仓结束。
- 最终结果必须包含盈利和亏损统计。

## 2. Milestone 0：设计文档落地

目标：完成项目初始化规划和文档对齐。

任务：

- 创建新项目目录 `global-cycle-investor-sim/`。
- 编写 README。
- 编写 CLAUDE.md。
- 编写需求梳理。
- 编写结构化头脑风暴。
- 编写 design spec。
- 编写 development plan。
- 编写内容治理与脱敏规范。
- 编写 MVP 验收标准。

验收：

- 文档覆盖至少 20 年周期、提前清仓、六市场、八板块、评分、事件、盈利/亏损统计、结局、复盘要求。
- 文档明确不接真实行情、不展示真实股票、不构成投资建议。
- 结局可能性不少于 10 个。

## 3. Milestone 1：项目脚手架

目标：建立可运行前端项目。

任务：

- 使用 Vite 初始化 React + TypeScript。
- 安装 Tailwind CSS。
- 配置 Vitest。
- 配置 lint。
- 创建基础目录：`src/app`、`src/pages`、`src/features`、`src/domain`、`src/data`、`src/shared`、`tests`。
- 创建基础 App 页面，显示产品标题与免责声明。

预期命令：

```bash
pnpm install
pnpm dev
pnpm build
pnpm test
pnpm lint
```

验收：

- `pnpm dev` 能启动本地页面。
- `pnpm build` 能生成构建产物。
- 初始测试命令可运行。

## 4. Milestone 2：静态内容与类型模型

目标：把游戏世界配置为可维护数据。

任务：

- 定义 TypeScript 类型：`GameState`、`YearScenario`、`Market`、`Sector`、`Allocation`、`EventCard`、`YearResult`、`LiquidationCheck`、`GameStats`、`Ending`。
- 定义 Zod schema。
- 创建六大市场数据。
- 创建八大板块数据。
- 创建至少 20 年宏观周期脚本。
- 创建事件卡池。
- 创建不少于 10 个结局定义，建议首版 12-14 个。
- 创建清仓线、初始本金、存档 key 等常量。

验收：

- 数据通过 schema 校验。
- 前端可读取并展示当前年度场景。
- 用户可见内容不含真实股票名或 ticker。
- 结局数据不少于 10 条。

## 5. Milestone 3：纯规则引擎

目标：实现可测试的核心模拟逻辑。

任务：

- 实现 allocation validation。
- 实现市场匹配评分。
- 实现板块匹配评分。
- 实现事件影响计算。
- 实现年度收益、年度盈利/亏损金额和回撤估算。
- 实现评分变化。
- 实现清仓检查：资产清仓线、深度回撤、连续集中、风险事件裸露。
- 实现盈利/亏损统计计算。
- 实现不少于 10 个结局的判定。
- 实现年度复盘生成。

单元测试：

- 分散配置比单一市场梭哈有更高风控评分。
- 宽松/复苏周期中成长/科技配置相对占优。
- 滞胀/衰退周期中防御板块相对抗跌。
- 高仓位集中配置在错误周期下显著增加回撤。
- 资产跌破清仓线会触发提前清仓。
- 盈利/亏损统计正确计算盈利年份、亏损年份、总盈亏、年化收益、最好/最差年份。
- ending selection 覆盖不少于 10 种结局。
- luck/event 不会扭转明显错误策略的长期结果。

验收：

- 规则引擎无需 UI 即可测试。
- 所有核心规则测试通过。
- 年度结果包含可解释 review items。
- 清仓结果包含清仓原因。
- 结局结果包含完整统计。

## 6. Milestone 4：最小网页闭环

目标：实现可玩的一局至少 20 年流程，并支持提前清仓。

任务：

- Home Page：开始/继续/免责声明，说明至少 20 年周期和提前清仓可能性。
- Annual Brief Page：展示宏观周期、政策、风险偏好、提示。
- Allocation Page：模板选择、仓位、市场权重、板块权重。
- Settlement Page：展示收益、盈利/亏损金额、资产、回撤、评分、事件、清仓检查。
- Review Page：展示复盘反馈并进入下一年。
- Ending Page：展示最终结局、提前结束原因和完整盈利/亏损统计。
- 使用 localStorage 保存和恢复 `GameState`。

验收：

- 玩家能完成完整至少 20 年。
- 高风险失败路径可以提前清仓。
- 页面不会卡在中间状态。
- 重开和继续游戏可用。
- 多种典型策略得到不同结局。
- 结局页展示盈利和亏损统计。

## 7. Milestone 5：体验与内容打磨

目标：让 MVP 更清晰、更像教育产品。

任务：

- 优化页面布局和响应式样式。
- 加强术语解释。
- 控制每屏信息量，避免 20 年流程疲劳。
- 补齐事件和复盘文案。
- 添加最终学习总结。
- 清仓页文案强调风险教育，而不是惩罚感。
- 进行内容脱敏审查。

验收：

- 首页和结算页显示免责声明。
- 所有术语有白话解释。
- 内容审查通过。
- 提前清仓原因清晰可解释。

## 8. Milestone 6：验证

目标：确认 MVP 可运行、可解释、无明显逻辑错误。

命令验证：

```bash
pnpm lint
pnpm test
pnpm build
```

人工验证：

1. 至少 20 年保守分散型策略。
2. 至少 20 年激进成长型策略。
3. 单市场/单板块梭哈策略，验证可能提前清仓。
4. 至少 20 年长期现金观望策略。
5. 前期亏损、后期改善策略。
6. 曾经高收益、后期大回撤策略。

每局检查：

- 年度结果是否符合宏观逻辑。
- 风控评分是否符合集中/分散行为。
- 清仓是否有明确触发依据。
- 结局是否符合长期行为。
- 盈利/亏损统计是否准确。
- 复盘是否能解释原因。

内容验证：

- 搜索真实股票名、ticker、logo 引用。
- 检查是否存在真实投资建议表述。
- 检查免责声明可见。

## 9. 延后事项

MVP 通过后再考虑：

- 个股精选模式。
- 随机种子和自由模式。
- 更多事件池。
- 图表增强。
- 后端存档。
- 排行榜。
- 部署上线。
